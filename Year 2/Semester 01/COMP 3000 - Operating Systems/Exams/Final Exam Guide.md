# Chapter 1 -- OS Intro
- Interrupts -- what they are, stages of handling
- Kernel-vs-User modes -- what each can/cannot do and why
- Computer Architecture basics (device drivers, bus, DMA...)
- Multi-programming vs multi-processing

### **Interrupts**
**What an interrupt is:**  
A hardware-generated signal telling the CPU: _"Stop what you're doing and run this handler."_

**Stages of Interrupt Handling**
1. Device triggers interrupt line.
2. CPU finishes current instruction.
3. CPU switches to kernel mode.
4. Program counter + registers saved.
5. Interrupt handler runs.
6. Handler clears interrupt.
7. CPU restores context and returns to user process.

**Types**
- Hardware interrupts (I/O device finished work)
- Software interrupts (exceptions, traps)

### **Kernel vs User Mode**
**User mode:**
- Can NOT directly access hardware
- Can NOT modify page tables
- Must use system calls

**Kernel mode:**
- Can access hardware & memory
- Runs interrupt handlers
- Performs scheduling, memory management

**Why two modes?**  
→ Protection & stability (users can't crash the machine).

**How do we switch modes?**
Only three ways:
1. **Hardware Interrupt** (device signals the CPU → enters kernel mode)
2. **Exception** (divide by zero, page fault → kernel mode)
3. **System Call (trap)**
    - User program intentionally executes an instruction like `int 0x80`, `syscall`, or `trap`
    - CPU automatically switches to kernel mode and jumps to OS handler.

The CPU always switches back to user mode when returning from the handler.

### **The kernel _is_ the operating system (OS) core.**
It _is_ the OS’s low-level component that:
- manages hardware
- manages processes
- manages memory
- handles system calls
- handles interrupts
- schedules the CPU
- manages filesystems
- controls security/protection


### **Architecture Basics**
- **Device Driver:** Software that controls a specific piece of hardware.
- **Bus:** Communication path (PCIe, USB).
- **DMA (Direct Memory Access):** A hardware feature that allows certain devices (disk controllers, network cards, GPUs) to copy data **directly into RAM** without involving the CPU for each byte.

### **Multiprogramming vs Multiprocessing**
- **Multiprogramming:** OS switches between **multiple programs** to keep CPU busy (single CPU).
- **Multiprocessing:** System has **multiple CPUs/cores**.

### **System Calls & Traps**
**Trap:**  
Software-generated interrupt used to enter kernel mode.

**System call vs Interrupt**
A **system call** is a synchronous trap generated by a user program to request OS services.  
An **interrupt** is an asynchronous event generated by hardware devices to notify the CPU of external events.  
Both cause control to enter the kernel, but system calls are intentional and predictable, whereas interrupts can occur at any time.

| System Call                 | Interrupt               |
| --------------------------- | ----------------------- |
| Intentional by user program | External/hardware event |
| Uses trap instruction       | Comes from device       |
| Synchronous                 | Asynchronous            |

### **OS Architectures**
**Monolithic**
- Very fast (all in kernel)  
    – Hard to maintain/debug

**Layered**
**Advantages:**
- Modularity & clear separation of responsibilities
- Easier debugging (problem isolated to a layer)
- Easier maintenance and enhancement
- Improves security (layers enforce boundaries)

**Disadvantages:**
- Hard to define proper layers
- Overhead: each layer adds processing/time
- Reduced performance compared to monolithic designs
```
+-------------------------+   ← Layer 5 (User programs)
|    Application Layer    |
+-------------------------+
|    High-Level OS APIs   |   ← Layer 4
+-------------------------+
|   File System, I/O Mgmt |   ← Layer 3
+-------------------------+
| Memory Mgmt, Scheduling |   ← Layer 2
+-------------------------+
| Hardware Interaction    |   ← Layer 1
+-------------------------+
|        Hardware         |   ← Layer 0
```


**Microkernel**
- Minimal kernel → services in user space  
    – Slow due to message passing overhead

### **Page Replacement Algorithms**
##### **FIFO (First-In, First-Out)**
**How it works**
- Maintain a queue of pages in the order they were loaded.
- When a page fault occurs:  
    → Remove the page that has been in memory **the longest**, regardless of use.

 **Strengths**
- Very simple to implement.
- Low overhead.

**Weaknesses**
- Does NOT consider how often or how recently a page is used.
- Can evict pages that are heavily used.
- Suffers from **Belady’s Anomaly** (more frames can cause _more_ faults).

**Note**
FIFO is based on **load time**, NOT access time.



##### **Optimal (OPT / MIN Algorithm)**
**How it works**
- Replace the page that will **not be used for the longest time in the future**.
- Requires future knowledge → impossible in real systems.

**Strengths**
- Produces the **minimum possible** number of page faults.
- Used as a benchmark to compare real algorithms.

**Weaknesses**
- Not implementable.
- Only theoretical or used in simulations.

**Note**
“minimum page faults” → **use Optimal**.



##### **LRU (Least Recently Used)**
**How it works**
- Replace the page that has **not been used for the longest time in the past**.
- Uses the idea that the past predicts the future.

**Implementation (conceptually)**
- Time-stamp each access OR
- Move page to MRU end of a list

**Strengths**
- Generally performs well.
- No Belady’s anomaly (stack algorithm property).

**Weaknesses**
- Expensive to implement exactly.
- Needs hardware support (access bits, counters) or approximations.



##### **Second Chance Algorithm (a FIFO improvement)**
**How it works**
- Works like FIFO, _but_ checks the **Referenced (Use) bit**:
    - If R = 0 → evict the page immediately.
    - If R = 1 → give the page a “second chance”:
        - Clear R bit to 0
        - Move page to the back of queue
        - Continue scanning

**Strengths**
- Simple approximation of LRU.
- Avoids evicting recently used pages.

**Weaknesses**
- Scanning can be slow if many R bits are set.
- Not as accurate as LRU.

**Note**
Second Chance is FIFO + R-bit, not LRU itself.



##### **Clock Algorithm (improved Second Chance)**
**How it works**
Same idea as Second Chance, but implemented using a **circular buffer** and a **clock hand**.

Algorithm:
1. Look at page pointed to by clock hand.
2. If R = 0 → evict it.
3. If R = 1 → set R = 0 and move clock hand forward.
4. Repeat until you find a victim.

**Strengths**
- Very efficient.
- Widely implemented.
- O(1) replacement per page (on average).

**Weaknesses**
- Still only approximates LRU.
- Can behave poorly if many pages are used frequently.

**Note**
CLOCK **reads and clears R-bit**, but DOES NOT set it. Hardware sets R-bit on access.


##### **Frequency-based Algorithms (LFU, MFU)**
 **LFU (Least Frequently Used)**
- Replace the page that has **been used the least number of times total**.

**Strengths**
- Captures long-term use frequency.

**Weaknesses**
- A page heavily used early but not recently may still stay forever.
- Requires counters.
- Can stick with outdated information unless counters decay.


**MFU (Most Frequently Used)**
- Replace the page used **most often**.
- Based on assumption:  
    “If a page has been used a lot, maybe it won’t be used again soon.”

Rarely used in practice.

**Note**
LFU ≠ LRU  
LFU cares about **how many times**; LRU cares about **how recently**.
LFU and MFU do not look into the future. Only up to what we've seen so far in RAM.


### **Frame Allocation**
- Fixed
- Proportional
- Global vs Local replacement
- Thrashing & Working Set


### **Speedups**
- Larger pages → fewer page faults
- TLB reach → TLB entries × page size
- Good locality improves performance

### **File Concepts & File Control Block (FCB)**
**What is a File?**
- A **logical, named collection** of related data stored on **non-volatile storage**.
- OS abstracts physical storage → provides a **uniform file interface**.
- File = **bytes / records / lines**, meaning defined by the creator.

### **Common File Attributes**
(Stored inside the directory entry or in a metadata structure)
- **Name** – human readable.
- **Identifier** – unique internal ID.
- **Type** – text, binary, executable, etc.
- **Location** – pointer(s) on disk.
- **Size** – current & possible max.
- **Protection** – permissions.
- **Timestamps** – create, modify, access times.

### **File Control Block (FCB)**
The **on-disk metadata structure** describing a file.  
Contains:
- File attributes
- Disk block pointers
- File size
- Reference count (in Unix inodes)
- Protection info

**In Unix:** FCB = **inode**  
**In Windows:** FCB ≈ **MFT entry**


### **File Access – Hardware vs Logical**
**Hardware-Level Access**
- Disks operate in **fixed-size blocks** (e.g. 512B, 4KB).
- All I/O is done in entire blocks.
- OS must **pack/unpack logical records** into blocks.

### **Logical Access Methods**
**(A) Sequential Access**
- Like reading a tape.
- **read_next()**, **write_next()**
- File pointer automatically moves forward.
- Most common (text files, logs).

**(B) Direct/Random Access**
- File viewed as numbered fixed-size blocks.
- **read(n)** reads block n directly.
- Useful for databases, indexed files.

**(C) Indexed Access**
- An **index** maps keys → block locations.
- Example exam systems:
    - ISAM
    - Multi-level indexes


# **Directory Structures**
Directory = **symbol table** mapping names → FCBs (inodes).  
Supports ops: search, create, delete, list, rename, traversal.


### Flat (Single-Level) Directory
```
/
 file1
 file2
 file3
```

**Pros**
- Simple to implement.

**Cons**
- Name collisions → every file must have a unique name.
- Not scalable for multi-user systems.


### Two Level Directory
```
MFD (Master File Directory)
 ├── userA → UFD_A
 ├── userB → UFD_B
```
Each user gets their **own directory**.

**Pros**
- Solves filename collisions.
- Basic isolation between users.

**Cons**
- Still poor for sharing between users.
- Requires specifying paths like `/userA/file`.



### Tree-Structured Directories
```
/
 ├── home/
 │    ├── user1/
 │    ├── user2/
 ├── bin/
 ├── etc/

```

**Pros**
- Natural, hierarchical organization.
- Supports absolute & relative paths.
- Easy navigation, scalable.

**Cons**
- Harder to maintain if many nested dirs.
- Sharing subtrees requires special mechanisms (links).

**Deletion Policy**
- Delete only empty dirs (safe but tedious), **OR**
- Recursive delete (dangerous but convenient).



### **Acyclic Graph Directory Structure**
Allows **shared files or directories** without cycles.
```
/home/user1/shared → (link)
                ↑
              /home/user2/shared

```

**Pros**
- True file sharing.
- No duplicated data → consistent updates.

**Must avoid cycles:**
- **Hard links only to files, not directories** (UNIX rule)
- OR OS must detect cycles (expensive)

**Deletion handling:**
- Use **reference counting**.
- Delete file only when count drops to 0.


### **General Graph Directory Structure**
Allows cycles (rare, almost never used).

**Problems:**
- Traversal must avoid infinite loops → requires:
    - Mark & sweep
    - Garbage collection
- Reference counts fail with cycles → memory leaks



### Hard Links vs Soft(Symbolic) Links

| Feature                    | Hard Link                                       | Soft (Symbolic) Link     |
| -------------------------- | ----------------------------------------------- | ------------------------ |
| Points to                  | inode / FCB                                     | path name                |
| Works for                  | files only                                      | files + directories      |
| If target deleted          | data remains until last hard link removed       | link becomes dangling    |
| Creates cycles?            | Not allowed (Unix forbids directory hard links) | Yes, possible            |
| Reference count increment? | Yes                                             | No (points to name only) |

**Hard link = another name for same file.**
**Soft link = pointer to pathname; can break.**

### **FS Layered Structure**
A file system is implemented in **layers**, each building on the lower ones:

**(1) I/O Control**
- Device drivers + interrupt handlers.
- Translate OS requests → hardware commands (“read block 123”).
- Manages interaction with storage device.

**(2) Basic File System (Block I/O Subsystem)**
- Reads/writes **logical blocks**.
- Manages buffer cache + I/O scheduling.
- Allocates buffers before performing block transfers.

**(3) File-Organization Module**
- Maps **files → logical blocks → physical blocks**.
- Includes **free-space manager** (tracks unallocated blocks).

**(4) Logical File System**
- Manages **metadata**.
- Handles:
    - Directory structures
    - File attributes
    - Security and permissions
    - File Control Blocks (FCBs / inodes)


### **Kernel (In-Memory) Structures Supporting FS Operations**
These exist **only while the FS is mounted** and speed up operations.

**Mount Table**
- One entry per mounted filesystem.
- Tracks:
    - Mount location
    - Device info
    - Superblock (volume metadata)

**System-Wide Open File Table**
- One entry per **open file in the entire OS**.
- Contains:
    - Copy of FCB/inode
    - File offset
    - File mode (read/write)
    - Count of processes referencing this file

**Per-Process Open File Table**
- One entry per open file **per process**.
- Points into **system-wide** open file table.
- Provides the file descriptor returned by open().


### **Directory Implementation**
**A. Linear List**
- List of (filename, pointer-to-FCB).
- **Pros:** simple.
- **Cons:** O(n) search, slow for large dirs.
- Improvement: **sorted list** or **tree** to reduce search time.

**B. Hash Table**
- Hash(filename) → index → pointer into linear list.
- **Very fast lookups.**
- Requires collision handling (chaining or probing).
- Resize issues exist (must rehash).

**C. Trees (e.g., balanced trees)**
- Alternative to sorted list.
- Fast search, insert, delete.


### **Block Allocation Methods**
❗ Differentiation Reminder:
- **Management data** stored separately (e.g., inodes, FAT tables).
- **File data** stored in regular data blocks

**1. Contiguous Allocation**
- File = single continuous region of blocks.
- Directory stores **start block + length**.

**Pros**
- Best sequential and direct access performance.
**Cons**
- **External fragmentation**
- Must know file size in advance.
- Hard to grow file.

Variant: **Extents**
- Allocate file in **chunks** ("extents").
- Reduces fragmentation and allows growth.

**2. Linked Allocation**
- File stored as linked list of blocks.
- Directory stores pointer to **first + last** block.

**Pros:**
- No external fragmentation.
- Easy growth.

**Cons:**
- **No direct access** (must traverse list).
- Pointer overhead.
- Reliability issues (lost pointer breaks file).

### Variant: **FAT (File Allocation Table)**
- FAT stores entire linked-list structure in one table.
- Directory only stores "first block".
- Faster random access because FAT is cached.

## **3. Indexed Allocation (General)**
- Each file has an **index block** holding array of block pointers.
- Directory stores pointer to index block.

**Pros**
- Supports **direct access**, no fragmentation.

**Cons**
- Index block overhead (especially for small files).
- Single-level index may be too small.


### **UNIX / Linux: The Inode Structure**
Inode stores:
- 12 direct pointers
- 1 single indirect
- 1 double indirect
- 1 triple indirect

Allows files to:
- Access small data quickly (direct blocks).
- Scale to very large sizes (multi-level indirect).


#### **Free Space Management**
**1. Bitmap (Bit Vector)**
- 1 bit per block:
    - **1 = free**, **0 = allocated**
- Easy to find N consecutive free blocks.
- Must fit in memory or be partially cached.

Systems often use **two bitmaps**:
- One for **inodes**
- One for **data blocks**


 **2. Linked List**
- Each free block points to the next.
- Simple but slow to traverse.

### **Grouping**
- A free block stores addresses of N more free blocks.
- Faster to retrieve many free blocks at once.

### **Counting**
- Track free space as **(start block, count)** pairs.
- Efficient when free blocks occur in contiguous runs.
- Similar to extents.


### **FS Performance Enhancements**
**1. Buffer Cache**
- Cached blocks or pages in memory.
- Unified buffer cache (Linux, Solaris, Windows):
    - Same cache used for memory-mapped I/O + read/write calls
    - Avoids double caching
    - Increase performance

**2. Free-Behind & Read-Ahead**
Used for **sequential access** optimization:

 **Free-Behind**
- Drop previous page/block once next block is accessed.

**Read-Ahead**
- Read next block(s) before needed.

**3. TRIM (For SSD/NVM)**
- SSDs must erase blocks before reuse.
- TRIM informs device which blocks are no longer valid.    
- Helps avoid "write cliff" where SSD slows dramatically.


### **Recovery + WAFL + Journaling**
**A. FSCK (Consistency Checker)**
- After a crash, scans metadata to repair:
    - Directory entries
    - Free block lists
    - Inodes
- Slow for large filesystems.
- Needed when the FS is not journaling.

**B. Journaling File Systems (from TEP Chapter 43 + textbook)**
**Idea:**
All metadata updates written to a **log** before being applied.

Steps:
1. Write transaction to log (journal).
2. Commit.
3. Apply changes to main FS.
4. Mark transaction as complete.

**Crash Recovery:**
- Reapply committed transactions.
- Discard uncommitted ones.

**Meta-journaling**
- Only logs **metadata**, not data blocks.
- Faster, less overhead (ext3 default mode).

**C. WAFL (Write Anywhere File Layout)**
Key Concepts:
- **Copy-on-write**: never overwrite blocks.
- **Snapshots**:
    - Created by copying root inode.
    - Only changed blocks allocated → snapshots are cheap.
- **Free-block map** tracks which snapshots reference a block.
- Enables:
    - Efficient replication
    - Versioning
    - Fast crash recovery

### WAFL advantages:
- Great for random writes (NAS environment).
- Uses NVRAM for stable logging.
- No need for fsck-style recovery.

# Unix File System

#### **Inodes**
An inode stores **metadata** :
- File size
- Block pointers
- Permissions
- Ownership
- Timestamps

#### **Data Blocks**
These hold the **actual file contents**.
If the disk has no free data blocks:
- You CAN still create a file **(allocate inode)**
- But you CANNOT write data into that file (no free blocks)

#### **Free space in UNIX FS is split into two pools**
- Free **inodes**
- Free **data blocks**
They are separate.  
You can run out of one before the other.

#### **`lseek()`**
`lseek(fd, offset, SEEK_SET)` **only changes the file offset inside the kernel**.  
It does **not**:
- write data
- allocate blocks
- change file size

Unless the next write goes beyond EOF, file size stays unchanged.
